# Odin Hub Operational Center Lock v7.1

## Purpose
Odin Hub is the Windows operational center of Odin Agent Shell. It is not just a dashboard. It is the canonical place where runtime entry, app pairing, model posture, QIRC posture, seed packs, pattern mines, runtime packs, handoffs, candidate review, recovery and support visibility converge.

## Core principle
If an operation affects Odin's local AI work boundary, it should be visible or routable through Odin Hub. This prevents tool sprawl, runtime sprawl and interpretation drift.

## Hub surfaces
- Home: current posture, runtime mode, active work, model availability, safety alerts.
- Apps: paired apps, Caller Manifest, Permission Cards, App-owned Apply boundary.
- Work Capsules: app/work context, active seeds, candidates, traces.
- Capability Packs: loaded runtime/capability slices.
- Seed Packs: app seed packs, compiled seed profiles, conflicts.
- Pattern Mines: registered mines, flow packs, pattern spines.
- Flow Packs: domain/workflow flow templates.
- Work Atoms: active atom graphs, budgets, skipped/executed status.
- QIRC Gold Spine: channels, hot windows, admissibility, ring radar.
- Models: local/remote worker cards, model routes, resource posture.
- Runtime Packs: manifest, validation state, rollback target.
- Handoffs: Thor/Y/Mjölnir packets, review gates, return candidates.
- Candidate Canvas: semantic diffs, candidate branches, merge/reject.
- Why Trace: route, seed, gate, model, candidate explanations.
- Diagnostics: doctor, health, events, support bundle, redaction preview.

## Operational commands
Start, stop, restart, doctor, safe-mode, compile-runtime-pack, validate-pack, compile-seed-pack, ingest-pattern-mine, export-support-bundle, purge-cache, rotate-trace, rollback-pack, revalidate-app, revoke-app, view-why-trace.

## Non-authority boundary
Odin Hub does not make app decisions. It does not apply app state. It does not send external messages. It does not accept model claims. It displays, routes, validates and prepares candidate work.

## Q Hub pattern relation
The hub pattern consolidates runtime entry, low-memory posture, QIRC posture, review/recovery visibility and bounded operational control into one operational center. Odin Hub applies that pattern to universal LLM work while preserving app sovereignty.

## Required DoD
Odin Hub implementation must provide a first-class product map, panel routing contracts, permission-aware visibility, support bundle export path, safe-mode UX, pack rollback UX, app revocation UX, and tests that panel actions cannot bypass apply boundary or final gate.
