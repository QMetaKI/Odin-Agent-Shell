## 2. Executive Definition

Odin v7.1.1 is a local-first, app-sovereign, candidate-only Small-Model Performance OS and Universal Semantic Work Kernel. It accepts bounded Universal Work from apps, coordinates work through QIRC/Semantic Bus and deterministic precompute, distills context, builds worklets and slots, creates ModelWorkPackets, routes through the smallest sufficient worker, uses 3B/7B/Hybrid model roles, runs critics and gates, emits Candidate Artifacts with Candidate DNA and receipts, and leaves all apply/state/external-send authority with the host app.

## 4. Central Function: Small-Model Performance OS

The main v7.1.1 metric is how much quality, speed, safety, boundedness, traceability, and app-usefulness Odin extracts from small local models. 3B and 7B/8B models must become more useful inside Odin than outside Odin because Odin supplies structure, not because Odin pretends a small model is larger. “AI without AI” means deterministic precompute, cache, rules, slots, contracts, critics, and gates do useful work before model work. Larger models are escalation, not identity. Remote is explicit only. Model output is projection until gated into a Candidate Artifact.

## 9. Universal Work Kernel

Universal Work is the primary input contract for every AI-like job entering Odin. It exists so apps never send vague prompts or hidden authority into the runtime; they send a bounded work object that can be checked, routed, traced, and returned as candidates.

Required Universal Work components:

- `input_artifacts`: typed references or payloads supplied under manifest policy.
- `transformation_verb`: the semantic operation requested, such as rewrite, summarize, review, extract, plan, compare, or classify.
- `output_contract`: the app-safe candidate shape Odin is allowed to return.
- `constraints`: style, length, privacy, latency, model, cost, source-use, and refusal constraints.
- `boundary`: candidate-only, no-apply, no-send, no-app-state-mutation, and claim/evidence constraints.
- `model_policy`: deterministic/no-model preference, allowed local tiers, remote explicit policy, and fallback rules.

Universal Work lifecycle:

1. `received`
2. `binding_checked`
3. `artifact_validated`
4. `verb_resolved`
5. `output_contract_checked`
6. `model_policy_checked`
7. `compiled`
8. `ready_for_context`
9. `blocked`
10. `needs_context`
11. `cannot_safely_complete`

Failure modes include missing binding, unknown artifact type, forbidden verb, output contract blocked, `candidate_only` false, direct apply intent, external send intent, privacy class denied, and claim boundary missing. These failures should return bounded rejection or context-request candidates instead of falling through to a model.

Small-model relevance: Universal Work gives small models bounded jobs, not vague prompts. It reduces ambiguity before routing, narrows the output contract, and allows deterministic checks to remove work the model should not perform.

QIRC/Semantic Bus relevance: Universal Work opens work batches, emits work lifecycle events, and anchors later context, slot, model, critic, candidate, and receipt events. Compile is not app apply. Validation is not model quality proof.

## 10. App Manifest, Binding, and App-Owned Apply

The Caller Manifest is Odin's first app boundary. It declares what an app is allowed to ask Odin to do before any single request is accepted. Binding is the per-request authority envelope that freezes the caller, request, manifest reference, privacy class, allowed outputs, forbidden actions, and claim boundary for that work item.

Manifest controls:

- allowed artifact types
- allowed verbs
- allowed output types
- blocked task classes
- model policy
- privacy policy
- bus policy

Binding freezes:

- caller
- request
- manifest ref
- privacy class
- allowed outputs
- forbidden actions
- claim boundary

App-owned apply law:

- Odin may propose actions as Candidate Artifacts.
- The app owns the apply gate.
- The app owns state mutation.
- The app owns external sends.

Negative examples: Odin must not save app project files, must not send email/message/webhook, and must not silently mutate plugin, game, website, editor, or database state. A candidate may describe a proposed edit or action, but the host app decides whether and how reality changes.

Future repo-real completion evidence should include manifest validator receipts, binding validator receipts, forbidden route tests, and SDK no-apply/no-send tests. A validated binding is still not app apply and not external-send authority.

## 11. Artifact Types, Lenses, Verbs, Output Contracts

Artifact Type is the typed input/output family that tells Odin what kind of material it is handling. A Lens defines what matters and what can be ignored for the current task. A Verb is the semantic transformation requested. An Output Contract is the app-safe shape and candidate boundary that the response must satisfy.

This layer exists to prevent generic output. The lens narrows source use, style, risk, level of detail, and domain anchors before a small model sees the task. The output contract prevents an otherwise plausible answer from becoming an unbounded blob.

Examples:

- `markdown + rewrite + clarity → markdown_candidate`
- `code_excerpt + review → review_summary_candidate`
- `event_digest + summarize → action_card_candidate`

QIRC target events include `#lens.select`, `#contract.resolve`, and `#artifact.validate`. These events should trace selection and validation decisions without granting authority. Registry presence is not runtime behavior; a declared artifact, lens, verb, or contract still requires validator and flow receipts before implementation claims are made.

## 12. Context Distillery and Semantic Pressure Valve

Context Distillery converts app-provided and repo-local material into a bounded Context Capsule. It exists because small models fail when they receive too much raw context, irrelevant context, private context, or context without a task center.

Pipeline:

1. app event digest
2. selected artifacts
3. lens focus
4. relevance filter
5. moment frame
6. context capsule
7. slot context

Context Capsule fields:

- `task_center`
- `must_use`
- `must_not_use`
- `style/tone`
- `length/shape`
- `allowed`
- `forbidden`
- `claim boundary`
- `source refs`
- `omitted context`
- `open questions`
- `confidence`
- `privacy class`

Semantic Pressure Valve behavior: detect too much context, split into worklets, ask the app for context, route no-model/cache where possible, or block if privacy is denied. This is one of Odin's most important small-model amplifiers because it turns broad app context into small, specific, privacy-bounded work packets.

QIRC channels should include `#context.*`, `#precompute.*`, and `#trace.*` for distillation decisions, omissions, context requests, and capsule receipts. Future evidence should include a capsule builder, omission tests, privacy denial tests, and generic warning tests.

## 13. Worklet Graph, Slot Forge, and Gaptext

The Worklet Graph decomposes Universal Work into bounded nodes with dependency edges, critic/refine edges, and fallback edges. It exists so Odin can route, cache, retry, and critique small pieces instead of forcing one model call to solve everything.

Slot Forge turns each worklet into a slot contract. The slot controls input type, output schema, allowed route, token budget, forbidden claims, retry policy, and fallback policy. Slot engineering is primary.

Slot Law:

- prompt engineering is secondary
- slot engineering is primary
- slot forging is adaptive

Gaptext is the bounded model input derived from validated work, lens focus, context capsule, and slot contract. Gaptext is not the raw app request and not raw app state. It is a compact work surface for the selected worker.

Small-model power: this is where 3B becomes useful. A 3B model can extract, repair JSON, classify, critique, or draft a narrow candidate when the slot is precise. QIRC channels include `#worklet.*`, `#slot.*`, and `#gaptext.*`. Generated gaptext is not truth, and slot success is not app apply.

## 16. 3B / 7B / 3B+7B Hybrid Operating Model

The default operating model treats small and medium local models as role-specialized workers. The key roles are 3B scout, 7B writer, and hybrid scout/write/check/compose.

| Model role | Jobs | Should not do | Odin support |
|---|---|---|---|
| 3B router | choose candidate route, detect simple blocks | decide app reality or hidden escalation | route policy, manifest, binding |
| 3B extractor | pull facts, headings, fields, constraints | synthesize unsupported conclusions | lens and Context Capsule |
| 3B compressor | shorten context into slot facts | drop required evidence silently | omission log and pressure valve |
| 3B schema repair | repair candidate JSON shape | invent missing facts | output schema and Minicheck |
| 3B no-go critic | detect unsafe, impossible, or forbidden requests | make final app decision | boundary registry and Final Gate |
| 3B claim critic | flag claims needing evidence | certify truth | claim/evidence gate |
| 3B genericness critic | find vague/cliché output | judge taste as authority | anti-generic rubric |
| 7B writer | draft structured candidate content | apply to app state | slot contract and output contract |
| 7B synthesizer | combine extracted facts into candidate | override forbidden claims | Candidate DNA and critics |
| 7B reviewer | review candidate coherence | certify correctness | critic cascade |
| 7B style rewriter | stabilize tone and shape | ignore source constraints | Style Stabilizer and Taste Dials |
| hybrid scout/write/check/compose | split scout, write, check, compose roles | bypass Odin gates | Worklet Graph and Final Gate |

Hybrid path: 3B scouts, extracts, and checks; 7B writes and synthesizes; 3B or deterministic critics check; Odin composes and gates. This can beat raw 7B-only because there is less context chaos, clearer task shape, cheaper critics, better retry/fallback, stronger consistency, and candidate-only safety.

## 20. Style, Specificity, Taste, Anti-Generic Engine

Style is not prompt wording. In Odin, style is output contract + lens + seed/pattern + critic + stabilizer + taste dials. This exists because small models often become generic unless specificity is made contractual.

Anti-generic checks ask:

- could this apply to any project?
- did it use concrete anchors?
- did it avoid source details?
- is it cliché?
- is it too vague?

Taste dials:

- warmer/clearer
- shorter/more detailed
- bolder/safer
- more technical/simpler
- formal/looser
- specific/general

Small-model power improves when taste is contractual, not improvised. The Style Stabilizer should normalize output without inventing facts, weakening boundaries, or making app decisions.

## 21. Work Memory, Semantic Cache, Model Dojo, Scoreboard

Work Memory stores bounded operational learning that improves future slot selection and route decisions. It may store successful slot contract, preferred output shape, failed route, app feedback, boundary issue, and fallback success.

Work Memory does not store by default:

- raw private chats
- full app DBs
- secrets
- unnecessary personal context

Semantic Cache may accelerate repeated deterministic, lens, context, or candidate-shape work. Semantic Cache may never bypass Final Gate.

Model Dojo tests should cover JSON fill, German rewrite, summary, schema repair, style control, event digest, no-go detection, context compression, and genericness. Scoreboard tracks model/task fit, route quality, slot success, retry/fallback success, and related routing signals. Scoreboard is not model quality certification.
