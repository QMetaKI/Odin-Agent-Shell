## Canon cleanup rule

Root files must stay readable. They may summarize history, but must not become the full history. CHANGELOG owns chronological history. Canon entrypoints own the current build route.

## Windows build-ready rule

Windows build readiness means that Codex has enough specification to implement:

- process topology;
- local IPC;
- localhost API;
- model runner supervision;
- runtime pack loading;
- seed/pattern pack validation;
- Odin Hub panels;
- safe mode;
- rollback;
- diagnostics and support bundle redaction.

It does not mean host validation has occurred.

## Boundary preservation

Every public-repo file must preserve:

- GPL-2.0-only identity;
- candidate-only output;
- app-owned apply;
- no external send by Odin;
- no unvalidated runtime pack load;
- no model/agent authority promotion;
- no hidden execution;
- no claim acceptance without receipt.

## Codex rule

Codex must build from the real PR bundles. Internal PR tasks remain the fine-grained ladder. Real PR bundles are the reviewable implementation units.
