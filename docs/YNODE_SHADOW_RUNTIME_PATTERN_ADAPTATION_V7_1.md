# YNode Shadow Runtime Pattern Adaptation for Odin v7.1

## Purpose

YNode-prep is used here as a pattern mine for how a complex architecture should become buildable by Codex. Odin does not copy YNode identity. Odin copies the preparation discipline:

```text
large canon → shadow runtime → mechanical build map → fixtures → tests → real code
```

## What Odin adopts

Odin adopts these principles:

- code-near shadow modules;
- contract-to-function mapping;
- state-machine clarity;
- receipt-before-claim discipline;
- event-envelope discipline;
- model-worker non-authority;
- final-gate preservation;
- build order before implementation;
- negative fixtures before dangerous behavior.

## What Odin does not adopt

Odin does not import:

- YNode public mesh;
- phone mesh;
- LAN federation;
- hosted relays;
- YNode app runtime scope;
- multiplayer runtime ownership;
- broad network QIRC features;
- YNode branding or identity.

## Odin-specific adaptation

Odin’s Shadow Runtime is scoped to:

```text
Universal Work
Candidate Artifact
Internal Semantic Bus
Small Model Power Layer
Model Scale Ladder
App Bridge
Claim Boundary
Codex PR/Bundles
```

## Codex Implication

Codex should treat this adaptation as the strongest build instruction after the Master Specs. It exists specifically so Codex can build Odin without re-deriving the architecture from scratch.
