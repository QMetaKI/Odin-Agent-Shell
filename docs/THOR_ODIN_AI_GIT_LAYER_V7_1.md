# Thor+Odin AI-Git Layer v7.1

## Definition
Thor+Odin is an AI-Git layer for model and agent work. It makes AI work reviewable, diffable, branchable, mergeable, rejectable, gated and traceable.

## Mapping
- Intent or Handoff = working tree input.
- Candidate Artifact = commit candidate.
- Semantic Diff = AI diff.
- Candidate Branch = alternate route or answer variant.
- Candidate Tournament = merge strategy.
- Thor Review = pull request review.
- Why Trace = commit message plus blame surface.
- Claim Boundary = protected branch rule.
- Generated Gates = CI checks.
- App-owned Apply = merge outside Odin.

## Universal session
An AI-Git work session is created for any nontrivial model/agent interaction. It binds the worker, adapter, route, capability card, permission card, work capsule, candidate outputs, semantic diffs and review state into one traceable object.

## Required states
RECEIVED, BOUNDARY_CHECKED, PRECOMPUTED, DISPATCHED, RETURNED, REVIEWED, CANDIDATE_READY, APP_REVIEW_REQUIRED, REJECTED or EXPORTED_AS_CANDIDATE.

## Why this matters
Raw model systems tend to produce direct answers. AI-Git systems produce candidates with lineage. This is the central safety and quality distinction.


## Non-negotiable red lines
- Any model. Any agent. Same Odin boundary.
- No worker may become app authority.
- No worker may perform app apply.
- No worker may perform external send through Odin.
- No adapter may bypass Odin Final Gate.
- No remote worker may receive raw private context unless explicitly allowed by privacy class and caller policy.
- No tool-using agent may execute tools through Odin unless a Permission Card allows the exact candidate operation.
- All outputs remain Candidate Artifacts, Review Notes, Risk Notes, PatchPlan Candidates, Handoff Returns, or Receipt Candidates.
- Thor discipline remains candidate-only and kernel-bound.
- GPL-2.0-only is the repository license identity for Odin and expected sibling identity for Thor when distributed together.
