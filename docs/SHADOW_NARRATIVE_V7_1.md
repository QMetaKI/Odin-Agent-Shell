# Shadow Narrative v7.1

## Purpose
Shadow Narrative is the negative twin of Fairy DSL. Fairy DSL describes the blessed path, the desirable archetypal order and the human-readable narrative aorta. Shadow Narrative describes the likely failure path: authority drift, seed poisoning, claim fog, excessive helpfulness, model overconfidence, app-apply confusion, QIRC fanout, prompt-injection through seed packs, and runtime-pack overclaim.

This document is part of Odin Agent Shell v7.1 / v0.7.0 Shadow Narrative Loki Anti-Pattern Lock. It is specification, shadow-runtime, validation and Codex-build guidance only. It does not claim runtime execution, model inference, host validation, network operation, deployment, security certification, app apply, external send, or full implementation completeness.

Core invariant: Odin remains candidate-only. App state, apply, external sends and domain reality remain app-owned. Loki/Shadow Narrative may reveal risk and propose gates; it may not decide authority or bypass Odin Final Gate.


## Senior Reviewer Decision
APPROVED WITH CONSTRAINTS. The idea improves Odin only when it is typed, mapped, gate-linked and test-generating. It is rejected if implemented as poetic prose, a dark persona, an agent, a swarm, or a new authority layer.

## Non-Negotiable Mapping Rule
No Shadow Narrative may remain free prose. Every narrative node must map to at least one machine-readable anti-pattern entry and at least one gate or test. If a node cannot be mapped, it is a hole and must be disclosed in Candidate DNA.

## Example
Fairy path: the small helper carries only the basket it can safely carry.
Shadow path: the small helper carries the whole forest because praise confused weight with worth.
Machine mapping: small_worker_overload -> slot_context_too_large -> semantic_pressure_valve -> split_work_or_escalate.
