# Universal LLM Work Construct v7.1

## Purpose
Thor+Odin must work for Y-native apps, ordinary apps, local models, hosted models, coding agents, browser/research agents, IDE assistants, workflow agents, and future tool-using assistants. The universal construct does not mean Odin implements every domain. It means every domain is translated into the same safe work form.

## Core formula
```text
Any model.
Any agent.
Any app.
Any workflow.
Same Odin boundary.
Same candidate protocol.
Same trace.
Same app-owned apply.
```

## Universal work chain
```text
External input / user intent / app event / repo signal / agent proposal
→ Adapter Boundary
→ Capability Card
→ Permission Card
→ Universal Work or Thor Handoff
→ Odin Core / QIRC / DFAS / Seed Economy
→ Model-Agent Worker
→ Candidate Artifact / Handoff Return
→ Semantic Diff / Why Trace
→ Human or App Review
→ App-owned Apply outside Odin
```

## Native and non-native integration
Y-native apps use deep Odin Work Capsules and QIRC hot-window integration. Non-native apps use adapters, digest bridges, permission cards, and candidate renderers. Both paths converge into the same Universal Work Kernel and Handoff Compiler.

## Required worker constraints
Every worker receives a Capability Card, a Permission Card, a Work Capsule, a Return Contract, a Claim Boundary and a Why Trace requirement. Workers may produce candidates, evidence notes, risk notes, review notes or patchplan candidates. Workers may not accept claims, issue receipts, execute returned code, apply patches, mutate app state, send externally, or escalate authority.

## Universal use-case coverage
Odin may support chat, coding, research, writing, business planning, document systems, GameDev, WordPress/plugin work, app building, Knowledge OS, legal-boundary review, family/youth systems, workflow automation, support assistants, ceremony/wedding studio workflows, NPC/quest generation, repo audit, Codex handoff, prompt compilation, RAG, scheduled candidate work, remote model finishing and low-memory local work. Each domain is handled by a profile and adapter, not by giving the model authority.


## Non-negotiable red lines
- Any model. Any agent. Same Odin boundary.
- No worker may become app authority.
- No worker may perform app apply.
- No worker may perform external send through Odin.
- No adapter may bypass Odin Final Gate.
- No remote worker may receive raw private context unless explicitly allowed by privacy class and caller policy.
- No tool-using agent may execute tools through Odin unless a Permission Card allows the exact candidate operation.
- All outputs remain Candidate Artifacts, Review Notes, Risk Notes, PatchPlan Candidates, Handoff Returns, or Receipt Candidates.
- Thor discipline remains candidate-only and kernel-bound.
- GPL-2.0-only is the repository license identity for Odin and expected sibling identity for Thor when distributed together.


## Objective benefit
The construct improves quality by standardizing context, boundaries, routes, return contracts and review. It improves efficiency by avoiding unnecessary large model calls and allowing local pre-LLM work. It improves safety by making model/agent work candidate-only, traceable and rejectable.
